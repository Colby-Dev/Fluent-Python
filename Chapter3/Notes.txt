---- Dictionaries and Sets ----

Python dictionaries (dict) are used often in Python, either directly with dict {} or indirectly with Class and instance attributes with __builtins__.__dict__. Dictionaries are optimized, as they are hash tables in the code behind Python. 

Other built-in types based on has tables are set and frozenset. Sets are highly optimized by using operations like union, intersection, subset test, etc. These allow us to use algorithms to avoid nesting numerous loops.

> Unpacking Mappings

	PEP 448 introduced support for mapping unpackings in two ways. First applying ** to more than one argument call in a function. Second is using ** inside a dict literal, multiple times regardless of data type or structure. 

	def dump(**kwargs):
		return kwargs

	dump(**{'x': 1}, 'y':2, **{'x':4})
	>>> {'x': 4, 'y':2}

> Merging Mappings with | 

	Python 3.9 supports | and |= to merge mappings. These are also set union operators:

		>> d1 = {'a':1, 'b':4}
		>> d2 = {'a':3, 'b':7, 'c':100}

		>> d1 | d2
		>> print(d1)

		   {a: 3, b: 7, c:100}

	The new mapping of d1 shows the left join of d2 overwriting the original dict and adding any additional elements. The inplace function is the same syntax and can be done in a cleaner way: 

		>> d1
		{'a':1, 'b':3}
		>> d1 |= d2
		>> d1
		{'a':2, 'b':4, 'c':6}

> Standard API of Mapping Types

	The collections.abc module provides the Mapping and MutableMapping ABCs describing the interfaces of dict and similar types. They do however require that keys are hashable, since they use the basic dict type in their implementation. 

> What Is Hashable? 

	A hash is a series of numbers that are immutable as they do not change from existing objects. Only immutable objects like tuples are hashables since any mutable data structures can change. 

> Inserting or Updating Mutable Values

	When trying to access a key value d[k] when k isnt an existing value will throw an error. You can use d.get(k, default) to handle the error as it will return nothing. The better way of doing this though is by using a .setdefault method like in Example 3-5.

	(see Example 3-5)

	By using the following: 

		my_dict.setdefault(key, []).append(new_value)

	This is the same as: 

		if key not in my_dict: 
			my_dict[key] = []
		my_dict[key].append(new_value)

	The key difference between these two parts of code from EX3-5 is the first looks for the key value and replaces the value. The latter part is much slowing as it compares all keys and then needs to find and update the value again. 

> Automatic Handling of Missing Keys

	Somtimes it is convenient to have mappings that return imaginary values when a missing key is searched for. The two ways of doing this would be to use defualtdict and a dict subclass and add the __missing__ method. 

	--> Defaultdict: Another Take on Missing Keys

		Using defaultdict, default values can be created on demand when a 
		missing key is searched. Inizialiing default dict, it will take a 
		mutable data structure such as a list. If a missing key is searched, 
		a new key is created and a reference to that list. 

		Using defaultdict works by calling the special method __missing__ 

	--> The __missing__ Method

		The underlying way mappings deal with missing keys is the __missing__ 		     method. The dict class will call dict.__getitem__ which will 
		implicitly trigger the __missing__ method, whenever a key is not 
		found instead of raising a KeyError. 

		(see Example 3-8)

> Inconsistent Usage of __missing__ in the Standard Library

	dict subclass, 
	-------------
		A subclass of dict implmenting only __missing__ and no other method.

	collections.UserDict subclass,
	------------------------------
		A subclass of UserDict implementing only __missing__ and no other 
		method. The get method inherited from UserDict calls __getitem__. 
		This means __missing__ may be called to handle lookups.

	abc.Mapping subclass with the possible __getitem__,
	---------------------------------------------------
		A minimal subclass of abc.Mapping implementing __missing__ and the
		required abstract methods. This includes an implementation of 
		__getitem__ taht does not call __missing__. 

	abc.Mapping subclass with __getiem__ calling __missing__,
	---------------------------------------------------------
		A minimal subclass of abc.Mapping implementing __missing__ and the 
		required abtract methods, including __getitem__ that calls 
		__missing__.

	With the four implementations mentioned, if they implement __getitem__, get, 
	and __contains__, then you can make these methods use __missing__. The 
	Standard library has many ways of using __missing__ which all may have edge 
	case behavior.

> Variations of dict

	collections.OrderedDict,
	-----------------------

		In Python 3.6, the built-in dict will order keys by default. The 
		most common reason then to use OrderedDict is writing code taht is 
		backward compatible with earlier versions.

	collections.ChainMap,
	--------------------

		The chainmap instance holds a list of mappings that can be searched 
		individually. This instance orders the keys, then ends once it finds
		the key, which is very efficient. 

	collections.Counter,
	-------------------

		A mapping that holds an integer count for each key. Using the 
		.update method adds to the value per the key. The counter mapping 
		takes a string then creates a dict from the string, where the key 
		is the strings and the value is the amount of occurances.

	shelve.Shelf,
	-------------
		
		The shelve module is a persistent storage for a mapping of string 
		keys to Python objects that are pickle binary format. The keys in 
		this module must be string. This module is basically a database 
		that stores mappings as a key in the DB and value as serialized 
		pickle binaries. 

> Subclassing UserDict Instead of Dict

> Dictionary Views

	The dict instance methods .keys(), .values(), and .items() return instances of classes called dict_keys, dict_values, and dict_items. These methods are read only methods of the dict type. These are all very efficient as no copying is happening. These read only methods are called views. 

	A view object is a dynamic proxy and is immutable. The classes dict_keys, 
dict_values, and dict_items are internal and are not available via built-ins or any standard library module.

	The dict_values class is the simplest dictionary view it implements only the __len__, __iter__, and __reversed__ special methods. The other two classes, dict_keys, and dict_items also implement their own special methods. 

> Practical Consequences of How Dict Works

	The hash table implementation of dict is very efficient but does have some 
	important information:

	* Keys must be hashable and implement proper __has__ and __eq__ methods. 

	* Item access by key is very fast, since hashes are directly compared. 

	* Key ordering is preserved as a sife effect for more compact memory layout

	* Dicts have significantly more memory over head. A hash table needs needs 
	  more data per entry

	* To save memory, avoid creating instance attributes outside of the
	  __init__ method

> Set Theory

	Sets in python are a type that is immutable and are apart of the standard library, and are now built-ins. They are a collection of unique objects and are used to remove duplication. 

	Set elements must be hashable. The set type is not hashable, so you can't build a set with a nested set.
